What is Apache Airflow?

Understanding the platform and its role in data engineering

Apache Airflow is an open-source workflow orchestration platform that allows you to programmatically author, schedule, and monitor complex data pipelines. Originally created at Airbnb in 2014, it's now part of the Apache Software Foundation and used by 80,000+ organizations worldwide with over 30 million monthly downloads.

The core idea: instead of writing brittle shell scripts or using rigid GUI tools, you define your workflows as Python code. This gives you version control, testing, reusability, and the full power of Python's ecosystem.

🚀

Airflow 3.0 (released April 22, 2025) is the biggest release in Airflow's history — bringing a service-oriented architecture, a new React-based UI, DAG versioning, asset-driven scheduling, and a Task Execution API enabling tasks to run in any language or environment.

What Airflow is great for

🔄

ETL / ELT Pipelines

Extract, transform, load data across databases, warehouses, and APIs reliably.

🤖

MLOps Workflows

Schedule model training, evaluation, and deployment pipelines. 30%+ of users use Airflow for ML.

⏰

Scheduled Jobs

Replace cron jobs with dependency-aware, monitored, retryable tasks.

📊

Data Quality

Run validation checks, generate reports, and trigger alerts automatically.

What Airflow is NOT

  • ❌ Not a data streaming tool — use Kafka or Flink for real-time processing
  • ❌ Not a data storage solution — it orchestrates pipelines, doesn't hold your data
  • ❌ Not a compute engine — it delegates actual computation to other systems (Spark, dbt, Python, etc.)

Installation

Three methods: pip (local), Docker Compose (recommended), or Docker custom image

⚠️

Prerequisites: Python 3.10–3.14 (3.12 recommended), pip or uv, at least 4GB RAM. Windows users must use WSL2. Airflow 3.x officially supports pip and uv — Poetry/pip-tools are not supported.

Method 1 — pip (Local / Virtual Env)

The simplest way to get started on Linux/macOS. Uses constraint files to guarantee reproducibility.

bash 
# 1. Create and activate a virtual environment
python3 -m venv airflow_env
source airflow_env/bin/activate   # Windows WSL: same command

# 2. Set version variables
AIRFLOW_VERSION=3.0.2
PYTHON_VERSION="$(python -c 'import sys; print(f"{sys.version_info.major}.{sys.version_info.minor}")')"
CONSTRAINT_URL="https://raw.githubusercontent.com/apache/airflow/constraints-${AIRFLOW_VERSION}/constraints-${PYTHON_VERSION}.txt"

# 3. Install Airflow with constraints (ensures compatible dependency versions)
pip install "apache-airflow==${AIRFLOW_VERSION}" --constraint "${CONSTRAINT_URL}"

# 4. Initialize the database & create admin user
airflow db migrate
airflow users create \
  --username admin \
  --firstname Admin \
  --lastname User \
  --role Admin \
  --email admin@example.com \
  --password admin

# 5. Start the standalone server (for local dev — runs scheduler + webserver)
airflow standalone

Open http://localhost:8080 and log in with admin / admin.

Method 2 — Docker Compose (Recommended for teams & exploration)

Docker Compose spins up the full Airflow stack with Celery workers, Postgres, Redis, and the webserver in a single command. Requires Docker Desktop with at least 4–8 GB RAM allocated.

bash 
# 1. Download the official docker-compose.yaml
mkdir airflow-project && cd airflow-project
curl -LfO 'https://airflow.apache.org/docs/apache-airflow/3.0.2/docker-compose.yaml'

# 2. Create required directories
mkdir -p ./dags ./logs ./plugins ./config

# 3. Set your user ID (Linux/macOS) — avoids file permission issues
echo -e "AIRFLOW_UID=$(id -u)" > .env

# 4. Initialize the database
docker compose up airflow-init

# 5. Start all services
docker compose up -d

# Useful commands
docker compose ps          # check status
docker compose logs -f     # tail all logs
docker compose down        # stop everything
💡

Place your DAG files in the ./dags/ folder. Airflow will automatically detect and load them within a minute. No restart needed.

Method 3 — Custom Docker Image

When you need extra Python packages on top of the base Airflow image:

dockerfile 
# Dockerfile
FROM apache/airflow:3.0.2

# Switch to root for system packages
USER root
RUN apt-get update && apt-get install -y build-essential libssl-dev

# Back to airflow user for pip installs
USER airflow
COPY requirements.txt /requirements.txt
RUN pip install --no-cache-dir -r /requirements.txt
text — requirements.txt 
apache-airflow-providers-postgres==5.13.1
apache-airflow-providers-http==4.12.0
pandas==2.2.0
requests==2.31.0

Then in your docker-compose.yaml, set image: ${AIRFLOW_IMAGE_NAME:-apache/airflow:3.0.2} and run docker compose build before docker compose up.

Your First DAG

Hello World — writing and running your first workflow

A DAG (Directed Acyclic Graph) is how you define a workflow in Airflow. Each node in the graph is a task; edges define the execution order. Create this file in your dags/ folder:

python — dags/hello_world.py 
from datetime import datetime
import pendulum
from airflow.sdk import dag, task  # Airflow 3.x import path

@dag(
    dag_id="hello_world",
    schedule=None,               # trigger manually
    start_date=pendulum.datetime(2024, 1, 1, tz="UTC"),
    catchup=False,
    tags=["tutorial"],
    description="My first Airflow DAG",
)
def hello_world():
    """A simple example DAG to demonstrate Airflow basics."""

    @task()
    def greet():
        print("👋 Hello from Airflow 3.x!")
        return "Hello"

    @task()
    def world(message: str):
        print(f"🌍 World says: {message} World!")

    # Chain tasks: greet runs first, its return value passes to world
    world(greet())


# Instantiate the DAG
hello_world_dag = hello_world()
ℹ️

Airflow 3.x import change: Most core objects (dag, task, DAG) now come from airflow.sdk rather than airflow or airflow.decorators. Both still work during migration, but airflow.sdk is the new canonical path.

Now go to http://localhost:8080, find hello_world in the DAGs list, toggle it on, and click the ▶ "Trigger DAG" button. Watch it run in the Graph or Grid view!

DAGs — Deep Dive

Directed Acyclic Graphs: the heart of every Airflow workflow

A DAG defines the structure of your workflow — which tasks exist and in what order they run. It is Directed (tasks flow in one direction), Acyclic (no loops — a task can't depend on its own future output), and a Graph (nodes connected by edges).

DAG parameters

python 
from airflow.sdk import dag
import pendulum

@dag(
    dag_id="my_pipeline",          # unique identifier
    description="ETL pipeline",       # shown in UI
    schedule="@daily",              # cron or preset (see below)
    start_date=pendulum.datetime(2024, 1, 1, tz="UTC"),
    catchup=False,                   # don't backfill missed runs
    max_active_runs=1,             # max concurrent dag runs
    max_active_tasks=5,            # max tasks running at once
    default_args={                   # applied to all tasks
        "owner": "data-team",
        "retries": 2,
        "retry_delay": timedelta(minutes=5),
        "email_on_failure": True,
        "email": ["alerts@company.com"],
    },
    tags=["etl", "production"],
)
def my_pipeline():
    ...

DAG versioning (New in Airflow 3.x)

Airflow 3.0 introduced DAG versioning — the system now tracks changes to DAGs over time. When you modify a DAG's structure (add/remove tasks, change dependencies), Airflow creates a new version and keeps the history, so past runs still link to the correct DAG structure. This is visible in the UI under the DAG's version history tab.

Operators

Pre-built task templates — the building blocks of pipelines

An Operator is a template that defines a single unit of work. Airflow ships with 900+ operators across 70+ providers. There are three main categories:

⚡

Action Operators

Execute something: run Python code, a Bash command, an SQL query, or an HTTP request.

🔁

Transfer Operators

Move data between systems. E.g., S3 to Snowflake, PostgreSQL to BigQuery.

⏳

Sensor Operators

Wait for a condition before proceeding. E.g., file exists, HTTP endpoint returns 200.

Common built-in operators

BashOperator

python 
from airflow.operators.bash import BashOperator

run_etl_script = BashOperator(
    task_id="run_etl",
    bash_command="python /opt/scripts/etl.py --date {{ ds }}",
    # {{ ds }} is a Jinja template variable = execution date (YYYY-MM-DD)
)

PythonOperator (classic style)

python 
from airflow.operators.python import PythonOperator

def my_function(**context):
    execution_date = context["ds"]
    print(f"Running for: {execution_date}")
    return "result_value"

run_python = PythonOperator(
    task_id="process_data",
    python_callable=my_function,
    # op_kwargs={"custom_param": "value"}  # optional keyword args
)

EmailOperator

python 
from airflow.operators.email import EmailOperator

send_report = EmailOperator(
    task_id="send_daily_report",
    to=["team@company.com"],
    subject="Daily Report - {{ ds }}",
    html_content="<h1>Pipeline completed for {{ ds }}</h1>",
)

BranchPythonOperator — conditional logic

python 
from airflow.operators.python import BranchPythonOperator

def choose_path(**context):
    hour = context["logical_date"].hour
    if hour < 12:
        return "morning_task"   # returns task_id to run next
    else:
        return "evening_task"

branch = BranchPythonOperator(
    task_id="decide_path",
    python_callable=choose_path,
)

branch >> [morning_task, evening_task]

Tasks & Dependencies

Wiring tasks together to form a pipeline

Tasks are instances of operators. You control execution order using dependency operators.

Setting dependencies

python 
# >> means "runs before"
extract >> transform >> load

# Multiple dependencies (fan-out)
extract >> [transform_a, transform_b]

# Multiple upstream tasks (fan-in)
[transform_a, transform_b] >> load

# Or use set_downstream / set_upstream explicitly
extract.set_downstream(transform)
transform.set_upstream(extract)   # equivalent

# Complex diamond pattern
extract >> [transform_a, transform_b]
[transform_a, transform_b] >> validate >> load

Visualizing the dependency graph

→ validate → load

TaskFlow API

The modern Pythonic way to write DAGs — recommended in Airflow 3.x

The TaskFlow API (introduced in Airflow 2.0, fully matured in 3.x) uses Python decorators to define DAGs and tasks. It handles XCom passing, task ID generation, and dependency tracking automatically — resulting in dramatically cleaner code.

Classic vs TaskFlow: the same pipeline

python — ❌ Classic (verbose) 
from airflow import DAG
from airflow.operators.python import PythonOperator
import pendulum, json

def extract(**ctx):
    data = {"1001": 301.27, "1002": 433.87}
    ctx["ti"].xcom_push(key="data", value=json.dumps(data))

def transform(**ctx):
    raw = ctx["ti"].xcom_pull(key="data", task_ids="extract")
    data = json.loads(raw)
    total = sum(data.values())
    ctx["ti"].xcom_push(key="total", value=total)

with DAG("etl_classic", ...) as dag:
    t1 = PythonOperator(task_id="extract", python_callable=extract)
    t2 = PythonOperator(task_id="transform", python_callable=transform)
    t1 >> t2
python — ✅ TaskFlow API (clean) 
from airflow.sdk import dag, task
import pendulum

@dag(
    schedule="@daily",
    start_date=pendulum.datetime(2024, 1, 1, tz="UTC"),
    catchup=False,
)
def etl_taskflow():

    @task()
    def extract() -> dict:
        # Return value is automatically pushed to XCom
        return {"1001": 301.27, "1002": 433.87}

    @task()
    def transform(order_data: dict) -> float:
        # order_data is automatically pulled from upstream XCom
        return sum(order_data.values())

    @task()
    def load(total: float):
        print(f"Total revenue: ${total:.2f}")
        # Write to database, S3, etc.

    # Chain: Python function calls = task dependencies
    order_data = extract()
    total = transform(order_data)
    load(total)


etl_dag = etl_taskflow()

Accessing context variables in TaskFlow

python 
from airflow.models.taskinstance import TaskInstance
from airflow.models.dagrun import DagRun

@task()
def use_context(task_instance: TaskInstance, dag_run: DagRun):
    print(f"Run ID: {task_instance.run_id}")
    print(f"Execution date: {dag_run.logical_date}")

Task decorators (Airflow 3.x)

Sensors

Waiting for external events before proceeding

Sensors are special operators that keep polling until a condition is met. They're perfect for waiting on files, API responses, database records, or other DAG completions.

FileSensor — wait for a file to appear

python 
from airflow.sensors.filesystem import FileSensor

wait_for_data = FileSensor(
    task_id="wait_for_csv",
    filepath="/data/input/sales_{{ ds_nodash }}.csv",
    poke_interval=60,            # check every 60 seconds
    timeout=3600,               # fail after 1 hour
    mode="reschedule",          # ⭐ releases worker slot between pokes
)

wait_for_data >> process_file   # only runs after file appears
💡

Always use mode="reschedule" in production sensors. In poke mode (default) the worker process sleeps and holds its slot. In reschedule mode, the worker is freed between polls, which is much more efficient at scale.

ExternalTaskSensor — wait for another DAG

python 
from airflow.sensors.external_task import ExternalTaskSensor

wait_for_upstream = ExternalTaskSensor(
    task_id="wait_for_ingestion",
    external_dag_id="data_ingestion",
    external_task_id="validate",   # None = wait for whole DAG
    mode="reschedule",
    poke_interval=120,
)

Custom sensor with @task.sensor

python 
from airflow.sdk import dag, task
from airflow.sdk import PokeReturnValue
import requests

@dag(schedule=None, ...)
def api_workflow():

    @task.sensor(poke_interval=60, timeout=3600, mode="reschedule")
    def wait_for_api_ready() -> PokeReturnValue:
        resp = requests.get("https://api.example.com/status")
        is_ready = resp.json().get("status") == "ready"
        return PokeReturnValue(is_done=is_ready, xcom_value=resp.json())

    @task()
    def process(api_info: dict):
        print(f"API is ready: {api_info}")

    process(wait_for_api_ready())

XComs — Data Passing

Sharing small values between tasks

XCom (Cross-Communication) is Airflow's mechanism for passing small pieces of data between tasks. Values are stored in the metadata database, so keep them small — a result ID, a row count, a file path — not a full DataFrame.

⚠️

XCom size limits: SQLite up to 2GB, PostgreSQL up to 1GB, MySQL up to 64MB. For large data, save to S3/GCS and pass the path via XCom instead.

Automatic XCom with TaskFlow

With the @task decorator, any returned value is automatically pushed to XCom, and any function parameter is automatically pulled from the upstream task's XCom:

python 
@task()
def get_record_count() -> int:
    return 42500  # auto-pushed to XCom

@task()
def process_records(count: int):
    print(f"Processing {count} records")

# The int "42500" flows automatically from get_record_count → process_records
count = get_record_count()
process_records(count)

Manual XCom push/pull (classic style)

python 
def push_data(**context):
    context["ti"].xcom_push(key="file_path", value="/s3/bucket/output.csv")

def pull_data(**context):
    path = context["ti"].xcom_pull(
        key="file_path",
        task_ids="upload_task"
    )
    print(f"File available at: {path}")

Variables & Connections

Managing configuration and secrets securely

Variables

Airflow Variables are key-value pairs stored in the metadata database. Use them for configuration values that change between environments (dev/prod) without modifying DAG code.

python 
from airflow.models import Variable

# Get a variable (with optional default)
s3_bucket = Variable.get("s3_bucket_name", default_var="my-default-bucket")

# Get a JSON variable (auto-deserialized)
config = Variable.get("etl_config", deserialize_json=True)
batch_size = config["batch_size"]

# Set a variable programmatically
Variable.set("last_run_date", "2024-03-15")

You can also set variables from the CLI or UI: Admin → Variables.

Connections

Connections store credentials for external systems (databases, APIs, cloud services). This keeps secrets out of your DAG code. Define them via UI (Admin → Connections) or environment variables:

bash — Environment variable format 
# Format: AIRFLOW_CONN_{CONN_ID}=conn_type://user:pass@host:port/schema
export AIRFLOW_CONN_MY_POSTGRES=postgresql://user:password@localhost:5432/mydb
export AIRFLOW_CONN_MY_S3=aws://AKIAIOSFODNN7EXAMPLE:wJalrXUt@/
python — Using a connection in a hook 
from airflow.providers.postgres.hooks.postgres import PostgresHook

@task()
def query_database():
    pg_hook = PostgresHook(postgres_conn_id="my_postgres")
    records = pg_hook.get_records("SELECT id, name FROM users LIMIT 10")
    return records

Scheduling

When and how DAGs run

Airflow schedules are based on logical dates (formerly execution dates). When a DAG with schedule="@daily" runs at midnight on January 2nd, its logical_date is January 1st — Airflow processes the data interval that just ended.

ℹ️

catchup: If catchup=True (the default) and your start_date is 30 days ago, Airflow will schedule 30 daily runs when you first enable the DAG. Set catchup=False to only run from "now" forward — recommended for most use cases.

Jinja Templating in schedules & tasks

python 
# {{ ds }} = execution date as YYYY-MM-DD  e.g. "2024-01-15"
# {{ ds_nodash }} = "20240115"
# {{ ts }} = full ISO timestamp
# {{ logical_date }} = pendulum datetime object
# {{ next_ds }} = next execution date
# {{ dag.dag_id }} = the DAG's ID string

run_query = BashOperator(
    task_id="daily_query",
    bash_command="""
      psql -c "SELECT * FROM events
               WHERE date = '{{ ds }}'
               AND batch = '{{ run_id }}'"
    """,
)

Asset-Driven Scheduling

Event-driven workflows triggered by data readiness — new in Airflow 3.x

In Airflow 3.x, Assets (formerly called "Datasets" in 2.x) let you trigger DAGs based on data availability rather than time. A DAG can declare that it produces an asset, and another DAG declares it consumes it — the consumer runs automatically when the producer completes.

✨

Airflow 3.2 added Asset Partitioning — allowing granular partition-level triggers. A downstream DAG can wait for a specific partition of an asset (e.g. region=US, date=2024-01-15) before running.

python — Producer DAG 
from airflow.sdk import dag, task, Asset

# Define an asset — a logical data resource
sales_data = Asset("s3://my-bucket/sales/daily")

@dag(
    schedule="@daily",
    start_date=pendulum.datetime(2024, 1, 1, tz="UTC"),
)
def ingest_sales():

    @task(outlets=[sales_data])  # ← signals this task produces the asset
    def download_and_store():
        # Download from source, write to S3...
        print("Sales data written to S3")

    download_and_store()

ingest_sales()
python — Consumer DAG (triggered automatically) 
from airflow.sdk import dag, task, Asset

sales_data = Asset("s3://my-bucket/sales/daily")

@dag(
    schedule=[sales_data],  # ← run whenever sales_data is updated
    start_date=pendulum.datetime(2024, 1, 1, tz="UTC"),
)
def transform_sales():

    @task()
    def run_transformations():
        print("Sales data is ready — running transforms!")

    run_transformations()

transform_sales()

Monitoring & Alerting

Keeping your pipelines healthy in production

Email alerts on failure

Configure email in airflow.cfg or environment variables, then use default_args:

python 
default_args = {
    "owner": "data-team",
    "email": ["oncall@company.com"],
    "email_on_failure": True,
    "email_on_retry": False,
    "retries": 3,
    "retry_delay": timedelta(minutes=5),
    "retry_exponential_backoff": True,  # 5min, 10min, 20min...
}

on_failure_callback — custom alerting (Slack, PagerDuty)

python 
import requests

def alert_slack(context):
    dag_id = context["dag"].dag_id
    task_id = context["task_instance"].task_id
    log_url = context["task_instance"].log_url
    requests.post("https://hooks.slack.com/services/YOUR/WEBHOOK", json={
        "text": f"❌ Task failed: {dag_id}.{task_id}\nLogs: {log_url}"
    })

default_args = {
    "on_failure_callback": alert_slack,
    "on_retry_callback": alert_slack,
}

Deadline Alerts (Airflow 3.1+ replacement for SLAs)

Airflow 3.0 removed the old SLA feature and replaced it in 3.1 with Deadline Alerts. Set a deadline on a DAG run to receive a callback if it hasn't completed by that time:

python 
from datetime import timedelta

@dag(
    schedule="@daily",
    # Raise an alert if the DAG hasn't finished within 2 hours
    dagrun_timeout=timedelta(hours=2),
    on_failure_callback=alert_slack,
    ...
)

Hooks & Providers

Reusable connection interfaces to external systems

A Hook is a low-level interface to an external system that handles authentication and connection management. Operators use hooks internally. You can use hooks directly in tasks for fine-grained control.

PostgreSQL Hook

python 
from airflow.providers.postgres.hooks.postgres import PostgresHook

@task()
def run_sql():
    hook = PostgresHook(postgres_conn_id="prod_postgres")
    
    # Get a pandas DataFrame
    df = hook.get_pandas_df("SELECT * FROM orders WHERE date = '{{ ds }}'")
    
    # Run INSERT/UPDATE
    hook.run("UPDATE metrics SET processed = True WHERE date = %s",
             parameters=["2024-01-15"])
    
    return len(df)

HTTP Hook

python 
from airflow.providers.http.hooks.http import HttpHook

Real-World ETL Example

A complete production-style pipeline with all the pieces

Let's build a realistic daily ETL pipeline: fetch data from a REST API, validate it, store in PostgreSQL, and send a Slack notification.

python — dags/daily_sales_etl.py 
from __future__ import annotations
import pendulum
import requests
from datetime import timedelta

from airflow.sdk import dag, task, Asset
from airflow.providers.postgres.hooks.postgres import PostgresHook

# Define the output asset (for downstream DAGs)
daily_sales_asset = Asset("postgresql://prod/sales/daily")


@dag(
    dag_id="daily_sales_etl",
    description="Fetch, validate, and store daily sales data",
    schedule="30 6 * * *",  # 6:30 AM every day
    start_date=pendulum.datetime(2024, 1, 1, tz="UTC"),
    catchup=False,
    default_args={
        "owner": "data-engineering",
        "retries": 2,
        "retry_delay": timedelta(minutes=5),
        "retry_exponential_backoff": True,
    },
    tags=["sales", "etl", "production"],
)
def daily_sales_etl():

    @task()
    def extract(**context) -> list[dict]:
        """Fetch sales records from the REST API."""
        ds = context["ds"]
        resp = requests.get(
            f"https://api.sales.internal/records?date={ds}",
            headers={"Authorization": "Bearer {{ var.value.api_token }}"},
            timeout=30,
        )
        resp.raise_for_status()
        records = resp.json()["data"]
        print(f"Extracted {len(records)} records for {ds}")
        return records

    @task()
    def validate(records: list[dict]) -> list[dict]:
        """Basic data quality checks."""
        required_fields = {"id", "amount", "customer_id", "product_id"}
        valid = []
        errors = 0
        for r in records:
            if required_fields.issubset(r.keys()) and r["amount"] > 0:
                valid.append(r)
            else:
                errors += 1
        if errors / max(len(records), 1) > 0.05:  # fail if >5% invalid
            raise ValueError(f"Too many invalid records: {errors}")
        print(f"Validation passed: {len(valid)} valid, {errors} skipped")
        return valid

    @task(outlets=[daily_sales_asset])  # signals asset production
    def load(records: list[dict], **context) -> int:
        """Upsert records into PostgreSQL."""
        ds = context["ds"]
        hook = PostgresHook(postgres_conn_id="prod_postgres")
        
        hook.run("DELETE FROM sales WHERE date = %s", parameters=[ds])
        
        rows = [(r["id"], r["amount"], r["customer_id"], ds) for r in records]
        hook.insert_rows(
            "sales",
            rows,
            target_fields=["id", "amount", "customer_id", "date"],
        )
        print(f"Loaded {len(rows)} records into sales table")
        return len(rows)

    @task()
    def notify(record_count: int, **context):
        """Send Slack success notification."""
        requests.post("https://hooks.slack.com/services/YOUR/WEBHOOK", json={
            "text": (
                f"✅ *Daily Sales ETL* completed\n"
                f"• Date: {context['ds']}\n"
                f"• Records: {record_count:,}\n"
                f"• Status: Success"
            )
        })

    # Wire the pipeline together
    raw_data = extract()
    clean_data = validate(raw_data)
    count = load(clean_data)
    notify(count)


# Instantiate the DAG
sales_etl = daily_sales_etl()
extract → validate → load → notify

Best Practices

Writing maintainable, production-grade DAGs

DAG authoring

  • Use catchup=False by default unless you specifically need backfilling
  • Set start_date as a fixed date, never datetime.now() — that would create unpredictable schedules
  • Keep tasks atomic — each task should do one thing. If a task fails, it should be safe to retry
  • Avoid top-level code in DAG files — imports and database connections at module level slow the Scheduler's DAG parsing loop
  • Use TaskFlow API for new DAGs — it's cleaner, handles XComs automatically, and is the direction Airflow 3.x is heading
🚫

Never do this: Don't put database connections or HTTP calls at the top level of your DAG file. Every DAG file is parsed by the Scheduler every 30 seconds. Slow top-level code will degrade your entire Airflow installation.

Performance

  • Use mode="reschedule" for all sensors in production
  • Set max_active_runs to prevent too many concurrent DAG runs
  • Use pool to limit concurrent tasks that share a resource (DB connections, API rate limits)
  • For Celery setups, keep tasks short and parallelizable — avoid single tasks that run for hours

Testing your DAGs

bash 
# Verify DAG parses without errors
airflow dags test daily_sales_etl

# Test a single task without running the full DAG
airflow tasks test daily_sales_etl extract 2024-01-15

# List all DAGs (check for import errors)
airflow dags list

# Check next scheduled run times
airflow dags next-execution daily_sales_etl

Pools — rate limiting resource usage

bash + python 
# Create a pool (UI: Admin → Pools, or CLI)
airflow pools set db_pool 5 "Limit concurrent DB operations"

    ...

What's New in Airflow 3.x

The landmark changes in the biggest release in Airflow's history

🏗️

Service-Oriented Architecture

New Task Execution API (AIP-72) enables tasks to run in remote environments with better isolation. API Server now handles all Airflow operations.

⚛️

New React UI

Fully rebuilt frontend — faster, more responsive, and modernized with asset lineage graphs and better dark mode support.

📦

DAG Versioning

Structural changes to a DAG now create a new version. Historical runs link to the DAG structure that was in place when they ran.

🎯

Assets (formerly Datasets)

Renamed and enhanced. Asset-driven scheduling with full lineage tracking. Airflow 3.2 adds partition-level asset triggers.

🌐

Edge Executor

Run tasks at the edge — distributed, event-driven workflows across geographically distributed environments (AIP-69).

🔭

OpenTelemetry

First-class OpenTelemetry traces for distributed tracing across task executions. Integrate with Jaeger, Datadog, or any OTel backend.

⏭️

Task Decorators

@skip_if and @run_if decorators let you conditionally skip tasks based on runtime conditions without boilerplate branching.

🛠️

airflow vs airflowctl

airflow is for local commands; airflowctl handles remote operations. Cleaner separation between dev and deployment.

Migration from Airflow 2.x

📋

Key breaking changes: Microsoft SQL Server support removed. AirflowTimeoutError no longer caught by default Exception. SLA feature removed (replaced by Deadline Alerts in 3.1). Import paths moved to airflow.sdk. Always review the official Upgrade Guide before migrating.
